[RECEIPT LLM - recommended models for RTX 5060 Ti 16GB]

extract_items_llm_only (vision):
  Best:    gemma3:12b        -- multimodal, best accuracy/speed tradeoff, fits easily in 16GB at Q4
  Backup:  qwen2.5vl:7b      -- faster, slightly lower accuracy, good for quick scans
  Avoid:   llama3.2-vision:11b -- slower than gemma3:12b with no accuracy gain on receipts

  Pros:
    - Single pipeline step, no intermediate error accumulation
    - Handles shadows, skew, and wrinkles better (sees spatial layout directly)
    - Faster end-to-end (~5-15s)
    - Less code/maintenance surface
  Cons:
    - Vision models are larger and require more VRAM than text-only models
    - Harder to debug — no intermediate OCR output to inspect
    - LLM may hallucinate items not present if image quality is very poor

extract_items_ocr (text only, no vision needed):
  Best:    qwen2.5:7b        -- excellent structured extraction from noisy text, very fast
  Backup:  gemma3:12b        -- overkill for text-only but works fine if already loaded
  Note:    OCR pipeline bottleneck is EasyOCR (cpu). Set gpu=True in receipt_ocr.py
           (_reader = easyocr.Reader(['en'], gpu=True)) to cut OCR time significantly.

  Pros:
    - Text-only LLM is smaller/faster than a vision model
    - OCR output is inspectable — easier to debug extraction errors
    - Perspective correction + deskew in receipt_ocr.py can improve results on bad photos
  Cons:
    - OCR errors compound into LLM errors (garbage in, garbage out)
    - Slower end-to-end (~30-60s with gpu=False, ~10-20s with gpu=True)
    - Two models to manage vs one
    - EasyOCR has no awareness of receipt structure — mangles multi-column price lines

Testing to see if we even need to cold boot...

Doesn't look like we need it

Ollama does cache image embeddings across separate calls, and it's well-documented.

The mechanism: Ollama computes a hash of the raw image bytes.
If you send the same image again while the model is still loaded,
it matches the cached KV slot and skips the forward pass for the image tokens entirely.
The vision encoder output is reused directly.

This is exactly why your two-pass approach works:

First pass processes the image fully, populating the KV cache with the image token embeddings
Final pass sends the same bytes → hash matches → image tokens are served from cache, and the model attends over them as if they were freshly computed
There's one critical caveat for your setup though: you're calling ollama.generate(model=VISION_MODEL, prompt='', keep_alive=0) to unload the model between
calls (llm.py:58 and llm.py:94). That wipes the cache entirely. The unload at line 94 (after the final pass) is fine, but the unload at line 58 (before the
first pass) means every run starts cold — which is intentional for the reload-to-fix-bad-state workaround.

Also worth noting: Ollama issue #11365 specifically flags that Gemma 3 has broken KV cache reuse due to its sliding window attention architecture,
so the caching benefit may not apply to gemma3:12b at all.
